## Project Summary

This section briefly describes what parts of the assignment were completed, the main technical decisions made during development, and possible improvements if more time were available.


## What Was Completed

### Must Requirements

All **Must** requirements were implemented.

#### M1 – Data Pipeline

A dataset of **50 unstructured property documents** was created with the help of an LLM.  
Each property listing contains realistic information such as:

- location
- price
- type
- size (sqm)
- number of rooms / bedrooms / bathrooms
- building details (floor, total floors, year built)
- property details (heating, parking, terrace, elevator, furnishing, condition)
- nearby transport
- description
- inspection notes
- legal information

The documents are stored as **.md files** and contain descriptive text rather than only structured fields in order to support meaningful semantic retrieval.

#### M2 – RAG Pipeline

The property search functionality is built using a **Retrieval-Augmented Generation (RAG)** pipeline.  
The pipeline converts unstructured property documents into embeddings and stores them in a vector database for semantic retrieval.

##### Vector Database

The system uses **ChromaDB** as the vector database.

Chroma was chosen because:

- it is lightweight and easy to run locally
- it requires no external infrastructure
- it integrates well with Python and embedding pipelines
- it supports persistent storage

The database is initialized using a **persistent client**, which stores the vector index on disk so that embeddings do not need to be recomputed every time the application starts.

##### Embedding Model

The system uses the **SentenceTransformers embedding model**:

```
all-MiniLM-L6-v2
```

This model was chosen because:

- it provides strong semantic similarity performance
- it is lightweight and fast
- it runs locally without requiring external APIs
- it is widely used for RAG pipelines and semantic search

Trade-offs:

- smaller models are faster but slightly less accurate than larger embedding models
- however, for this dataset size, the performance difference is negligible


#### Chunking Strategy

Property documents are stored as **Markdown files** and are chunked based on their section structure.

Instead of splitting by a fixed number of tokens or characters, the system splits documents at **Markdown section headings (`##`)**. Each section becomes an individual chunk.

Each `##` section is extracted and combined with the property header to create a **self-contained chunk**.

This strategy ensures that every chunk contains:

- the **property identifier**
- key metadata from the header
- a **focused section of descriptive information**

This approach was chosen for several reasons:

- property documents already have a **natural semantic structure**
- each section represents a **distinct type of information** (description, amenities, legal notes, etc.)
- chunks remain **short enough for accurate retrieval**
- each chunk is **self-contained**, allowing it to be interpreted without needing the full document

By prepending the property header to every chunk, the system ensures that retrieved chunks still contain important context such as:

- property name
- price
- size
- number of rooms

This improves the quality of responses generated by the RAG pipeline.

Additionally, each chunk is assigned a unique identifier:

```
{property_id}_chunk_{index}
```

Metadata is stored alongside each chunk:

```
property_id
```

This metadata allows the retrieval pipeline to later group results by property and avoid returning multiple chunks from the same listing.

#### Retrieval Process

The retrieval pipeline is designed to return **distinct relevant properties**, not just the top matching chunks.

Retrieval works in **two stages**:

- **Stage 1:** semantic similarity search over all chunks
- **Stage 2:** deduplication and ranking at the property level

This approach was chosen because the documents are chunked by Markdown sections, so a single property can produce multiple relevant chunks. Without deduplication, the results could be dominated by one property appearing multiple times instead of showing a diverse set of matching listings.

### ML Component – Property Price Prediction Model

A machine learning model was implemented to **predict the estimated market value of a property** based on structured features extracted from the property documents.  
The predicted value is used by the **Analytical Buyer Agent** to make more informed bidding decisions during the auction simulation.

---

#### Feature Extraction

Structured features are extracted from the property listings, including:

- location
- size (square meters)
- number of rooms
- selected property features (when available)

These values are parsed from the property documents and converted into a structured dataset used for model training.

---

#### Model Training

Two ensamble models were trained and evaluated:

- **Random Forest Regressor**
- **XGBoost Regressor**

Both models were trained to predict the **expected market price of a property** based on the extracted features.

#### Model Evaluation

The models were evaluated using standard regression metrics:

- **MAE (Mean Absolute Error)** – measures the average absolute difference between predicted and actual prices.
- **RMSE (Root Mean Squared Error)** – penalizes larger prediction errors more strongly.
- **R² (Coefficient of Determination)** – indicates how well the model explains the variance in the data.

After evaluating both models, **Random Forest** was selected as the final model because it produced better overall results across these metrics for the available dataset.

The model demonstrated:

- lower MAE
- lower RMSE
- higher R² score

compared to the XGBoost model.

Given the small dataset, Random Forest provided more stable performance and reduced risk of overfitting.

---

#### Model Serialization

The selected model was saved as a serialized file:

```
price_prediction_model.pkl
```

This allows the trained model to be reused during runtime without retraining.


#### Integration with Buyer Agent

At runtime, the model is loaded and used inside the **Analytical Buyer Agent strategy**.
The **Analytical Buyer Agent** uses the trained model to estimate the market value of a property before deciding whether to place a bid.

The bidding logic works as follows:

1. Extract structured features from the property listing.
2. Use the ML model to estimate the property's predicted price.
3. Compare the predicted value with the current auction bid.
4. Decide whether to bid.


If the predicted value is significantly higher than the current bid, the agent is more likely to continue bidding.  
If the predicted value is close to or below the current price, the agent may stop bidding.

This enables the Analytical Buyer Agent to make **data-driven bidding decisions instead of relying purely on heuristic strategies**.

#### Limitations

The dataset used for training is relatively small (generated property listings), so the model is primarily intended to demonstrate **ML integration within the agent-based system** rather than provide production-grade price predictions.

In a real-world scenario, the model would benefit from:

- larger historical property datasets
- richer location features
- market trend data
- property transaction history


## What can be improved with more time?

Several improvements could make the system more robust and closer to production quality.

### Stronger Auction Orchestration

The auction orchestrator currently manages the bidding rounds and basic validation rules. With more time, additional constraints and validation logic could be added, such as:

- enforcing stricter bidding rules
- validating agent budgets across multiple auctions

Expanding the orchestration logic would make the auction simulation more realistic and resilient.


### More Buyer Agents

Currently the auction includes a limited number of buyer agents.  
The system could be extended to support a larger and more diverse set of agents with different profiles, including:

- investors focused on rental yield
- long-term homeowners
- agents specialized in certain neighborhoods or property types

More agents would produce more dynamic and competitive auction simulations.


### Multi-Property Investment Strategy

Currently agents bid on **one property at a time**. A more realistic simulation would allow agents to:

- select **multiple properties of interest**
- allocate their budget across several auctions
- prioritize properties based on predicted value and preferences

Agents could first analyze the available listings and choose a subset of properties they want to compete for before the auctions begin.


### Auction Selection Strategy

Instead of manually selecting the property to auction, agents could first evaluate the available properties and choose which ones should be auctioned.

This would simulate a more realistic marketplace where multiple properties are available and buyers must decide which opportunities to pursue.


### Improved Retrieval and Data Quality

Further improvements could be made to the RAG pipeline, including:

- hybrid search combining semantic retrieval and structured filters
- domain-specific embedding models
- evaluation of retrieval performance
- fine-tuning the embedding model

Additionally, the system could benefit from larger and more realistic real estate datasets.

### Build a Web Scraper

To create a larger and more realistic dataset, the project could be extended with a web scraper that collects real property listings from public real estate websites.
This would improve the quality and diversity of the input data compared to a fully synthetic or manually created dataset and would make both the RAG pipeline and the price prediction model more realistic.
At the same time, this comes with an important limitation: some popular Bulgarian property websites contain duplicate, outdated, or misleading listings, which can reduce dataset quality. Because of this, a scraping pipeline would also need additional validation and cleaning steps.


### Web Interface

The current CLI interface is sufficient for demonstration purposes, but a web interface could significantly improve usability.

A web application could provide:

- interactive property search
- visual auction progress
- agent activity monitoring
- auction history visualization

### Enhanced Machine Learning Models

The price prediction model could be improved with:

- larger datasets
- historical transaction data
- more advanced models

This would allow buyer agents to make more accurate property valuations.
